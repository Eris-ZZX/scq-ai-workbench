# Part 1. PRD：产品需求文档

## 1. 产品定位

供应链质量部 AI 综合工作台是一个面向供应链质量部的统一 AI 应用平台，用于聚合业务运行、组织管理、人才发展、知识资产、AI 能力、工具技能和管理决策。

产品不是多个零散 AI 工具的集合，而是部门级统一工作台：

```text
统一入口
统一权限
统一数据底座
统一 AI 能力中心
统一工具技能层
统一审计与安全治理
```

## 2. 产品目标

| 目标 | 说明 |
|---|---|
| 业务结果可见 | 在“我的工作台”展示三条主业务线的结果情况、风险卡点和我的关联事项。 |
| 日常待办聚合 | 将待处理、待复核、待确认、待补资料、待关闭事项集中到个人首页。 |
| 专项能力沉淀 | 各角色工作台承载专项拓展功能，如 NPQ 二三级计划生成、PQE PFMEA、SQE 8D 初审。 |
| 知识可检索可复用 | 钉钉知识库作为知识源头，本地建立权限化索引和 RAG 检索。 |
| AI 输出可复核 | AI 做分类、汇总、判断建议、预警、生成，但关键结论必须人工确认。 |
| 管理决策可量化 | 提供业务、组织、人才三类管理看板，指标口径统一。 |
| 企业数据安全可控 | 按数据等级、权限、脱敏、模型路由、审计和应急机制控制数据风险。 |

## 3. 用户与角色

| SQE 工程师 | 供应商画像、8D 追踪、来料风险、供应模式差异、供应商改善闭环。 |
| PQE 工程师 | 量产异常、过程风险、改善立项、复发预警、节拍影响。 |
| EMS 工程师 | 代工厂异常、三地版本一致性、厂区差异、跨厂区同步。 |

## 5. 我的工作台需求

“我的工作台”是每个业务用户的日常首页，不承载复杂拓展工具。它根据角色展示：

- 三条主业务线结果情况。
- 我的业务待办。
- 我的复核事项。
- 我的风险提醒。
- 我的专属业务信息。
- 我的 AI 建议。
- 我的知识推荐。

专项拓展能力进入各角色工作台。

## 6. 标准 AI 调用链路

```text
用户请求
→ UserContext 识别
→ 功能权限和数据权限判断
→ 数据等级识别
→ 字段脱敏或上下文裁剪
→ Model Gateway 模型路由
→ AI 调用
→ JSON Schema / 输出格式校验
→ 置信度、不确定点和引用来源生成
→ 高风险场景进入人工复核
→ 审计记录写入 ModelInvocation / AuditEvent
→ 展示、生成文件或受控写回
```

## 8. 钉钉知识库联动

```text
钉钉知识库作为原始知识源
→ 同步知识库、目录、文档元数据、权限、更新时间、来源链接
→ 可下载内容进入本地解析、分块、向量化
→ 用户检索时先按本地角色权限 + 钉钉权限过滤
→ AI回答必须带钉钉原文链接和引用来源
```

关键规则：

- 本地索引不能绕过钉钉权限。
- 钉钉文档权限变化后，本地索引必须同步更新或失效。
- 跨知识库检索必须同时满足本地权限和钉钉权限。
- 过期、废止、版本冲突文档不得参与 AI 回答。

## 9. 企业数据安全

| 等级 | 数据示例 | 要求 |
|---|---|---|
| 普通 | 公开培训资料、通用方法论、无敏感字段模板 | 可进入知识库和 AI 问答。 |
| 内部 | SOP、检验标准、流程模板、一般复盘案例 | 仅限授权人员查看和调用 AI。 |
| 敏感 | 供应商表现、8D、批退记录、客户相关问题 | 默认限制导出和外发，必要时脱敏。 |
| 严格受控 | 客户保密资料、供应商处罚/评级、价格成本、重大事故、人员评价 | 默认不进入公网模型，优先本地或私有化模型。 |

控制要求：

- 业务对象、字段、知识文档必须标记敏感等级、责任人和可用范围。
- RAG、AI 总结、文件生成前必须做权限过滤。
- 客户、供应商、价格、人员评价、事故编号支持脱敏。
- 按数据等级选择公网、企业受控、本地或私有化模型。
- 所有工具必须注册、授权、审计。
- 敏感数据导出记录导出人、字段、脱敏状态、用途和审批状态。
- 记录 AI 输入摘要、检索来源、输出、工具调用、人工修改、最终动作。

## 10. 权限与治理

```text
RBAC：控制角色能进入哪些模块和功能
ABAC：控制用户能看到哪些数据、字段、知识和组件
```

权限维度：

- 角色。
- 业务线。
- 厂区。
- 供应商。
- 项目。
- 流程节点。
- 数据敏感等级。
- 动作权限：查看、编辑、导出、调用 AI、审批、写回系统。

权限申请链路：

```text
权限不足提示
→ 用户提交 AccessRequest
→ 系统带出申请对象、数据范围、敏感等级和建议有效期
→ owner / 管理员审批
→ 权限生效并记录有效期
→ 到期自动回收或进入复核
→ 全流程写入审计日志
```

## 11. 工作流原则

建议采用 LangGraph.js 承载状态化、可恢复、可人工中断/复核的业务闭环。

原则：

- 业务流程优先于 Agent 自主性。
- AI 只做辅助分类、汇总、判断建议、预警和生成。
- 关键质量结论、供应商处置、客户输出、人才评价必须人工复核。
- 所有关键节点保留状态、责任人、操作记录和 AI 输入输出。

## 13. SPEC 验收标准

| 验收项 | 测试方式 |
|---|---|
| 业务对象完整性 | 抽查 SQE、PQE、NPQ、EMS、QCM、实验室是否都能映射到统一业务对象。 |
| 主数据可关联 | 用供应商、物料、批次、异常单、8D、改善项做完整链路关联测试。 |
| 指标口径一致性 | 检查 MetricDefinition 中公式、维度、周期、数据源、负责人和版本。 |
| 权限矩阵 | 用不同角色账号验证菜单、数据、导出、AI 调用、审批权限差异。 |
| 数据安全 | 用敏感字段测试脱敏、模型路由、导出审批和审计记录。 |
| RAG 权限过滤 | 用户无法通过知识问答检索到无权限钉钉文档内容。 |
| 钉钉知识同步 | 文档新增、更新、失效、权限变化后，本地索引能同步或标记重建。 |
| AI 结构化输出 | 分类、汇总、8D 初审、异常分析输出必须符合 JSON Schema。 |
| 工具调用审计 | 网页浏览、文件生成、钉钉通知、数据查询均有审计记录。 |
| 人工复核 | 高风险动作不能直接完成，必须进入复核台。 |
| 组件注册 | 每个组件有唯一标识、数据源、参数 schema、权限策略、展示 schema 和刷新策略。 |
| 个人工作台 | 用户可复制模板、增删组件、拖拽布局、保存个人视图。 |
| 新功能扩展 | 新功能可通过组件注册加入组件库，并被个人工作台和团队看板引用。 |

## 14. 默认假设

- 所有核心能力优先本地代码实现，不强依赖 Dify。
- Dify 只作为可选参考、过渡工具或能力对比对象。
- 钉钉知识库作为知识源头，本地只做权限化索引、检索增强、AI 引用和审计。
- 钉钉身份、部门和知识库权限需要与本地用户、角色和数据范围同步映射。
- 网页浏览、文件生成、系统查询、钉钉联动等能力通过工具与技能层受控调用。
- 企业数据安全模块贯穿所有工作台、AI 能力、工具技能、知识库同步和系统集成。
- 敏感和严格受控数据默认不直接进入公网模型，必须经过权限判断、脱敏或模型路由控制。
- 权限设计依赖数据对象、字段口径、敏感等级和业务范围梳理结果。
- 首期以我的工作台、SQE、PQE、知识资产、人工复核为 MVP。
- 复杂工作流仅保留架构建议，具体流程图在业务 0 层完成后单独设计。
- 初期外部系统接口不齐时，允许使用 Excel/CSV、手工录入和 API mock 过渡。
- 工作台和看板采用模块化组件方案，预设角色工作台只是模板，不是用户可用视图的上限。
- 所有正式管理指标必须先进入 MetricDefinition，再被看板组件引用。
- 权限不足必须通过 AccessRequest 申请和审批，不允许线下绕过系统长期授权。

## 5. Main user workflow

1. NPQ 创建新产品导入项目。
2. 系统基于活动模板库一次性生成全阶段母任务和子任务。
3. NPQ 在母任务上维护计划完成时间，子任务默认继承。
4. 负责方进入项目活动页，按阶段筛选或按角色筛选自己的子任务。
5. 负责方更新子任务状态、交付件或完成说明，必要时填写阻塞说明。
6. 子任务完成后，母任务自动汇总完成率和风险标记。
7. 全部子任务完成后，母任务进入待 NPQ 关闭。
8. NPQ 核查母任务闭环情况，手动确认关闭。
9. 管理层看板持续展示项目总体进度、阶段进度、风险和角色按时完成率。

| 修改子任务计划时间例外 | 子任务负责方和 NPQ 可修改。 |
| 关闭母任务 | 仅 NPQ。 |
| 查看管理层看板 | 管理层、NPQ、授权角色。 |

## 9. Metrics rules

母任务自动汇总：

- 子任务全未开始：母任务未开始。
- 任一子任务进行中或已提交：母任务进行中。
- 全部子任务已完成：母任务待 NPQ 关闭。
- NPQ 确认后：母任务已关闭。
- 任一子任务阻塞或逾期：母任务同步显示风险标记。

逾期判断：

- 子任务优先使用自己的计划完成时间。
- 子任务未设置例外时间时，沿用母任务计划完成时间。
- 超过计划完成时间且未完成时，标记逾期。

角色子任务按时完成率：

- 分子：按时完成的子任务数量。
- 分母：截至当前日期应该完成的子任务数量。
- 判断口径：子任务 `已完成时间 <= 子任务计划完成时间` 为按时。
- 已到期但未完成的子任务计入分母，并视为未按时。
- 未到期子任务不计入分母。

## 10. Edge cases and risks

- Excel 中交付件为空：允许作为无需交付件任务，但提交时必须填写完成说明或确认备注。
- 同一项目任务名称跨阶段重复：不合并，按 `阶段 + 项目任务` 分别生成母任务。
- 同一母任务下同一三级计划由多个负责方执行：分别生成子任务。
- NPQ 代维护：必须进入 ActivityEvent，避免责任链丢失。
- 第一版不做项目当前阶段切换：阶段只作为分组和筛选维度。
- 第一版不做复杂审批流：审批、复核、确认负责人字段后续扩展。

# F3 build-ready 计划包（2026-06-11 更新）

## 1. App summary

F3 聚焦 **NPQ 活动流真实运转 + 配套后台底座完善**。F2 已能展示和更新活动任务，但距离真实项目管理系统仍有明显差距：模板仍偏硬编码，岗位角色未真实落地，项目启动缺少向导，执行者缺少我的待办，交付件缺少真实附件，阶段门和项目生命周期不足，后台配置和运行日志命名也不够业务化。

F3 的目标不是横向扩更多业务线，而是把 NPQ 新产品导入活动流打穿为一个可配置、可分派、可追踪、可复盘的真实业务系统。后台建设必须服务于这条流，包括模板中心、岗位角色、动作级权限、运行日志、站内提醒和项目阶段门。

## 2. Scope principles

- F3 聚焦 NPQ 活动流，不启动 SQE/PQE/知识库/AI 等新业务线。
- 后台底座只做到支撑 NPQ 活动流真实运转，不做全平台通用流程引擎。
- 模板从 Excel 初始数据升级为系统内结构化配置；F3 不做 Excel 导入/导出。
- 项目创建后依赖项目活动快照运行；后台模板更新默认不影响已创建项目。
- 一线执行体验优先：用户通过“我的待办/我的责任项”处理任务，而不是每天进项目里翻表。
- F4 候选需求提前记录，但不混入 F3 实现范围。

## 5. Main user workflow

1. admin 在模板中心维护多个内置 NPQ 活动模板。
2. admin 发布某模板的最新版本，供新项目创建使用。
3. admin 维护岗位字典和用户岗位。
4. NPQ 创建项目，进入 4 步项目向导。
5. NPQ 填写项目信息，选择某个内置模板。
6. 系统自动使用该模板的最新已发布版本。
7. NPQ 在向导中任命项目岗位负责人。
8. 系统预览将生成的阶段、母任务、子任务、责任岗位、负责人和交付件要求。
9. NPQ 确认后生成项目活动快照。
10. 项目进入当前 TR 阶段，系统生成我的待办和站内提醒。
11. 岗位负责人从我的待办进入子任务，填写说明或上传附件后完成任务。
12. NPQ 可退回不合格子任务，负责人在待办中重新处理。
13. NPQ 可对项目快照做有限调整，如新增任务、调整负责人、标记不涉及。
14. 当前阶段涉及母任务关闭且无阻塞后，NPQ 可通过阶段门。
15. 存在逾期、未完成例外或遗留风险时，NPQ 必须填写条件放行说明。
16. 阶段通过后自动进入下一 TR 阶段。
17. 管理层查看阶段时间线、剩余事项、逾期列表、完成率和角色按时率。

- 不涉及任务不进入我的待办。
- 不涉及任务不进入角色按时完成率分母。
- 不涉及任务不进入风险/阻塞统计。
- 不涉及任务只作为项目活动结构中的说明性记录展示。

按时完成率：

- 分子：已完成且完成时间 <= 计划完成时间的涉及子任务。
- 分母：截至当前日期应该完成的涉及子任务。
- 已到期但未完成的涉及子任务进入分母，并视为未按时。
- 未到期任务不进入分母。

## 12. Edge cases and risks

- 模板没有发布版本：项目向导不能选择该模板。
- 项目岗位未任命：必须在向导中提示，不应静默生成无负责人任务。
- 后台模板更新后：已创建项目不自动变更。
- 项目内新增任务：必须标记来源为项目内新增，便于和模板任务区分。
- 不涉及任务恢复涉及：恢复后重新进入待办、逾期和统计计算。
- 附件删除：必须保留删除记录，不应从动态记录中消失。
- NPQ 退回已完成任务：任务完成率、母任务状态和阶段门判断需要重新计算。
- 条件放行：必须记录说明，避免阶段门变成随意跳转。
- 批量操作：每个被影响任务都要能追溯变更，不能只记一条笼统记录。
- 权限矩阵配置错误：至少 admin 保留兜底权限，避免系统自锁。

## 13. Acceptance criteria

1. 后台可以创建、复制、编辑、发布、停用 NPQ 活动模板。
2. 后台可以维护模板的阶段、母任务、子任务三级结构。
3. 子任务模板可以配置责任岗位、交付件、附件/说明要求、默认计划偏移、排序和启停。
4. 项目创建向导可以按 4 步完成项目启动。
5. 项目创建时只能选择某内置模板的最新已发布版本。
6. 项目创建后生成项目活动快照，后续后台模板更新不影响该项目。
7. 后台可以维护岗位，一个用户只能绑定一个岗位。
8. 项目中可以为岗位任命负责人，一个用户可参与多个项目。
9. 我的待办能展示当前用户负责的待处理、已退回、即将到期、已逾期和待补交付件任务。
10. 子任务可以上传、下载、删除轻量附件，删除必须留痕。
11. 子任务状态只有未开始、进行中、已退回、已完成。
12. NPQ 可以退回子任务，退回必须填写原因并进入负责人待办。
13. NPQ 可以在项目内新增/调整活动快照，并用不涉及标记项目例外。
14. 不涉及任务不进入任何计算。
15. NPQ/admin 可以执行轻量批量操作，所有批量操作留痕。
16. 系统能生成站内提醒和我的待办角标。
17. 后台运行日志包含操作日志、系统错误、接口请求。
18. 业务详情显示动态记录。
19. 项目支持 TR 阶段门，阶段通过后自动进入下一阶段。
20. 阶段切换满足关闭 + 无阻塞 + 条件放行说明规则。
21. 项目详情显示阶段时间线、当前阶段剩余事项、即将到期/逾期任务列表。
22. 权限矩阵可以控制 NPQ 活动流关键动作。
23. lint、typecheck、unit/integration test、build 和本地冒烟测试通过。

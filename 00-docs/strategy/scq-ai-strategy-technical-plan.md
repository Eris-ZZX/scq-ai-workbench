# 质量工程师日常整合工作台 - PRD & SPEC

> 本文件是项目唯一正式 PRD & SPEC。它描述产品需求和技术规格，不承载 VibeTracker 计划、实施 backlog、完成记录或会话日志。
> 最新更新：2026-06-22。

# Part 1. PRD

## 1. 产品定位

质量工程师日常整合工作台是面向质量工程师和质量管理团队的统一工作入口。它不是单一的新产品活动系统，也不是多个零散 AI 工具的集合，而是质量工程师每天处理项目、待办、任务、证据、风险、协作和管理信息的整合型工作台。

产品定位可以概括为：

```text
统一入口
个人待办聚合
项目工作区
通用任务与证据链
角色化动作
运行记录与配置中心
后续 AI 能力预留
```

新产品活动 / NPQ 是当前已落地的一个业务模块，用于验证工作台的项目推进、任务流转、角色权限、证据留痕和阶段管理能力。它不能反向定义整个工作台的产品边界。

## 2. 产品目标

| 目标 | 说明 |
|---|---|
| 日常工作聚合 | 用户登录后能先看到自己要处理的任务、关注的项目、风险与下一步动作。 |
| 项目推进清晰 | 每个项目有统一工作区，展示阶段、项目活动、子任务、状态、历史和处理动作。 |
| 任务闭环可追踪 | 任务分派、处理、退回、关闭、不涉及、附件和代维护都有记录。 |
| 角色动作明确 | 不同角色看到不同操作；业务执行和管理查看边界清晰。 |
| 模块可扩展 | NPQ 只是第一个业务模块，后续 SQE、PQE、QCM、实验室等模块应复用工作台能力。 |
| 配置可维护 | 管理员可维护用户、角色、模板、项目结构和运行记录。 |
| AI 能力可接入 | 长期可接入分类、汇总、判断建议、检索推荐、报告生成等 AI 能力，但关键结论必须人工确认。 |

## 3. 用户与角色

### 3.1 当前已落地角色

| 角色 | 主要关注 | 当前边界 |
|---|---|---|
| NPQ | 项目创建、活动模板、岗位任命、项目推进、退回、关闭、阶段门。 | 当前 NPQ 模块的核心业务角色。 |
| PQE / SQE / FAE / RAM / QCM 等项目角色 | 处理被分配的项目任务，提交说明或附件，响应退回。 | 作为项目岗位负责人参与当前项目。 |
| 管理者 | 查看项目进度、风险、逾期、待关闭和整体状态。 | 默认偏只读，不参与具体业务处理。 |
| 管理员 | 维护用户、角色、模板、项目结构、权限和运行记录。 | 不作为业务处理人。 |

### 3.2 后续可扩展角色

| 角色 / 模块 | 可能扩展方向 |
|---|---|
| SQE | 供应商画像、8D 跟踪、来料风险、供应商改善闭环。 |
| PQE | 量产异常、过程风险、改善立项、复发预警。 |
| QCM | IQC / IPQC / FQC / OQC 执行数据、批次风险和标准依据。 |
| 实验室 | FAE 分析、可靠性评估、尺寸结果、ORT、标准评审输入。 |
| 部门负责人 | 多业务线风险、资源协调、管理看板和重点项目状态。 |

## 4. 信息架构

```text
前端业务界面
├─ 工作台首页
│  ├─ 我的待处理任务
│  ├─ 我的项目
│  ├─ 风险 / 逾期 / 阻塞 / 待关闭提示
│  └─ 快速进入项目工作区
├─ 项目工作区
│  ├─ 项目状态摘要
│  ├─ 阶段 / 项目活动 / 子任务结构
│  ├─ 任务详情与处理动作
│  └─ 当前任务历史 + 项目活动关键历史
├─ 业务模块
│  └─ NPQ / 新产品活动
├─ 配置中心
│  ├─ 用户与角色
│  ├─ 项目与活动模板
│  ├─ 权限矩阵
│  └─ 运行日志
└─ 后续模块入口
   ├─ SQE
   ├─ PQE
   ├─ QCM
   ├─ 实验室
   ├─ 知识管理
   └─ 管理看板
```

## 5. 工作台首页需求

工作台首页是用户登录后的主入口，不应该变成某一个业务模块的首页。

核心区域：

| 区域 | 需求 |
|---|---|
| 我的待处理任务 | 按统一优先级展示任务，支持查看全部、逾期、阻塞、退回、待确认关闭等状态。 |
| 我的项目 | 展示用户参与或关注的项目，包含当前阶段、进度、下一待处理、阻塞 / 逾期 / 待关闭标记。 |
| 快速跳转 | 点击待办直达项目工作区的具体任务；点击项目进入项目工作区并默认展开当前阶段。 |
| 角色化信息 | 不同角色看到与自己相关的数据，不显示无权限项目或动作。 |

优先级建议：

```text
阶段门卡住 > 阻塞 > 逾期 > 退回 > 待确认关闭 > 我的责任项
```

## 6. 项目工作区需求

项目工作区承载真实业务处理，而不是仅展示项目摘要。

| 区域 | 需求 |
|---|---|
| 项目摘要条 | 展示项目名称、当前阶段、整体进度、关键状态和风险提示。 |
| 左侧结构树 | 按阶段 / 项目活动 / 子任务展示，当前阶段展开，其他阶段可折叠。 |
| 详情面板 | 展示任务信息、负责人、计划时间、交付件要求、附件、完成说明、阻塞、退回原因。 |
| 处理动作 | 按角色展示可执行动作，如开始、完成、退回、关闭、不涉及、上传附件、代维护等。 |
| 历史记录 | 展示当前任务历史和所属项目活动关键历史，所有关键动作留痕。 |

## 7. 当前业务模块：NPQ / 新产品活动

NPQ 模块是当前首个业务模块，范围应保持小而专注。

已确立的业务能力：

| 能力 | 说明 |
|---|---|
| 活动模板 | 管理活动模板和阶段 / 项目活动 / 子任务结构。 |
| 项目创建 | 创建项目并生成项目活动快照。 |
| 岗位任命 | 项目内按岗位任命负责人，模板责任岗位映射到具体负责人。 |
| 项目活动快照 | 项目创建后独立运行，模板后续修改默认不影响已创建项目。 |
| 我的待办 | 展示当前用户负责的待处理、退回、逾期、待补交付件、待关闭任务。 |
| 附件与说明 | 子任务支持附件、完成说明和交付件要求。 |
| 退回与代维护 | NPQ 可退回子任务，也可按规则代维护并留痕。 |
| 不涉及 | 对项目例外项标记不涉及，不进入完成率和逾期等计算。 |
| 阶段门 | 支持 TR 阶段推进、条件放行和阶段记录。 |
| 运行日志 | 管理后台展示操作日志、系统错误和接口请求；业务详情展示动态记录。 |

NPQ 不应扩大的范围：

- 不把 NPQ 的阶段、权限、术语硬编码成全平台标准。
- 不在 NPQ 内实现所有质量业务。
- 不把审批、完整文件中心、复杂报表、外部通知、AI 风险生成等后续能力塞进当前模块。

## 8. 后续模块方向

后续模块进入前，必须先判断它复用哪些工作台能力，新增哪些业务专属能力。

| 方向 | 复用能力 | 专属能力示例 |
|---|---|---|
| SQE | 项目 / 任务 / 证据 / 动态记录 / 角色动作 | 供应商画像、8D 追踪、稽核清单、来料风险。 |
| PQE | 项目 / 任务 / 证据 / 报表入口 | 量产异常、PFMEA 辅助、改善优先级、复发识别。 |
| QCM | 任务 / 检验记录 / 标准依据 / 批次追踪 | IQC 综合判断、批次风险、检验标准匹配。 |
| 实验室 | 任务 / 附件 / 结论留痕 / 项目关联 | FAE 分析、可靠性评估、ORT、尺寸测量结论。 |
| 知识管理 | 权限、引用、检索、动态记录 | 知识同步、知识入库、相似案例推荐。 |

## 9. 暂不纳入当前版本

- 正式审批流。
- 交付件在线预览。
- 外部通知。
- 独立项目风险清单 / 完整风险管理。
- 模板发布审批。
- 完整文件中心。
- 复杂报表筛选 / 保存视图。
- 甘特图。
- 全平台通用权限引擎。
- AI 功能扩展。

## 10. PRD 验收标准

| 验收项 | 标准 |
|---|---|
| 产品定位 | 文档、导航和页面表达均体现“质量工程师日常整合工作台”，不把 NPQ 当成产品整体。 |
| 工作台首页 | 用户能看到自己的待处理任务和项目，并可快速进入项目工作区。 |
| 项目工作区 | 项目阶段、活动、子任务、详情、动作和历史记录能形成处理闭环。 |
| 角色差异 | NPQ、一线角色、管理者、管理员看到的数据和动作符合边界。 |
| 证据留痕 | 附件、说明、退回、关闭、不涉及、代维护和阶段动作都有记录。 |
| 模块边界 | NPQ 模块小而专注；后续能力以停车场或独立模块形式沉淀。 |
| 可维护性 | 管理后台能维护必要配置，运行日志能辅助排查。 |

# Part 2. SPEC

## 1. 技术基线

| 层 | 当前选择 |
|---|---|
| Fullstack | Next.js App Router + TypeScript |
| UI | Tailwind CSS + shadcn/ui |
| Database(dev) | SQLite |
| Database(prod path) | PostgreSQL + pgvector 预留 |
| ORM | Prisma |
| Auth | 自定义 JWT / 用户名密码，后续预留企业登录 |
| Runtime | 本地 Windows 开发，后续内网访问 / 内网服务器部署 |
| Test | Vitest / Testing Library / 本地冒烟 / Full Test |
| AI path | Model Gateway、RAG、LangGraph.js 作为长期预留 |

## 2. 系统架构

当前应用采用单体全栈架构，内部按“平台层、业务模块、后台配置”拆分边界。

```text
quality-workbench-app
├─ src/app
│  ├─ (auth)
│  ├─ (dashboard)
│  │  ├─ workbench
│  │  ├─ admin
│  │  └─ flows / npq
│  └─ api
├─ src/platform
│  ├─ auth
│  ├─ permissions
│  ├─ observability
│  └─ ui
├─ src/flows
│  └─ npq
├─ src/admin
├─ src/lib
└─ prisma
```

依赖原则：

```text
app -> platform / flows / admin
flows -> platform
admin -> platform
platform -> lib
flows 之间不直接依赖
admin 不承载业务处理逻辑
```

| 对象 | 作用 |
|---|---|
| User | 登录用户、基础身份和系统角色。 |
| PositionRole | 岗位 / 项目责任角色。 |
| UserPosition | 用户与岗位绑定。 |
| Project | 质量项目主体。 |
| ProjectMember / ProjectPositionAssignment | 项目成员和项目岗位任命。 |
| ActivityTemplateSet / ActivityTemplateVersion | 活动模板集和版本。 |
| ActivityTemplateStage / Parent / Child | 模板阶段、项目活动、子任务结构。 |
| ProjectActivity / ProjectTask | 项目内实际活动和任务快照。 |
| ActivityAttachment | 子任务附件证据。 |
| ActivityEvent | 业务动态记录。 |
| Notification | 站内提醒和待办角标。 |
| StageGateRecord | 阶段门通过、条件放行和阶段记录。 |
| NpqActionPermission | NPQ 模块动作级权限。 |
| ObservabilityEvent | 请求、错误、系统事件和运行日志。 |

## 4. 状态与计算规则

子任务状态：

```text
未开始
进行中
已退回
已完成
```

父级活动状态：

```text
未开始
进行中
待确认关闭
已关闭
```

不涉及规则：

- 不进入完成率。
- 不进入母任务关闭条件。
- 不进入逾期。
- 不进入我的待办。
- 不进入角色按时完成率分母。
- 不进入风险 / 阻塞统计。
- 仅作为项目活动结构中的说明性记录展示。

阶段门规则：

- 当前阶段原则上需要关闭涉及的项目活动。
- 当前阶段不能存在有效阻塞任务。
- 存在例外时必须填写条件放行说明。
- 阶段动作必须写入记录。

## 5. 权限规则

当前权限以项目参与、岗位任命和动作级权限结合为主。

| 动作 | NPQ | 当前责任角色 | 管理者 | 管理员 |
|---|---|---|---|---|
| 查看相关项目 | 是 | 被任命项目 | 可配置 | 是 |
| 处理自己的子任务 | 可代维护 | 是 | 否 | 否 |
| 上传附件 | 可代维护 | 是 | 否 | 否 |
| 退回子任务 | 是 | 否 | 否 | 可配置 |
| 关闭项目活动 | 是 | 否 | 否 | 可配置 |
| 标记不涉及 | 是 | 否 | 否 | 可配置 |
| 阶段通过 / 条件放行 | 是 | 否 | 否 | 可配置 |
| 维护模板和配置 | 可配置 | 否 | 否 | 是 |
| 查看运行日志 | 可配置 | 否 | 可配置 | 是 |

长期权限方向保留 RBAC + ABAC，但当前版本不建设全平台通用权限引擎。

## 6. API 设计原则

- API 必须先校验会话。
- 业务对象读取必须校验项目访问权限。
- 业务动作必须校验动作权限和对象状态。
- 写操作必须写入动态记录或运行日志。
- 列表接口默认限制分页和排序，避免一次性返回过大数据。
- 面向页面的数据聚合可以使用专用 query/helper，但不得把页面状态写死到业务模型。

核心 API 类型：

| 类型 | 示例 |
|---|---|
| Auth | 登录、登出、当前用户。 |
| Workbench | 我的待办、我的项目、项目推进摘要。 |
| Project | 项目 CRUD、项目成员、项目岗位任命。 |
| Activity | 项目活动结构、任务详情、状态动作、批量调整。 |
| Attachment | 上传、下载、删除和附件校验。 |
| Template | 模板集、模板版本、阶段 / 活动 / 子任务结构。 |
| Admin | 用户、岗位、权限矩阵、配置中心。 |
| Observability | 运行日志、系统错误、接口请求。 |

## 7. 可观测性与日志

运行日志面向管理员和排障场景，不作为业务用户的主页面。

| 日志类型 | 内容 |
|---|---|
| 操作日志 | 用户对项目、任务、模板、权限、附件、阶段门的关键变更。 |
| 系统错误 | 服务端异常、失败任务、堆栈摘要。 |
| 接口请求 | 关键 API 请求、耗时、状态码、traceId。 |
| 业务动态记录 | 项目、活动、子任务、附件和阶段门的业务历史。 |

要求：

- 每个关键 API 请求生成或传递 traceId。
- 业务写操作保留 actor、时间、对象、动作和摘要。
- 错误日志不能泄露敏感信息。

## 8. AI 能力预留

AI 能力不是当前已完成主线，但架构应保留接入点。

| 能力 | 预留用途 |
|---|---|
| 分类 | 异常、任务、资料、风险分类。 |
| 汇总 | 项目状态、任务历史、交付件、问题记录摘要。 |
| 判断建议 | 阶段评审、退回建议、风险提示。 |
| 检索推荐 | 相似案例、SOP、历史改善和模板推荐。 |
| 结构化生成 | 报告、会议纪要、检查清单和复盘草案。 |

AI 使用原则：

- AI 只给建议，不直接替代质量结论。
- 输出必须带依据、来源、不确定点和人工确认入口。
- 敏感数据进入模型前必须做权限判断、脱敏或模型路由。
- AI 调用必须审计。

## 9. 数据安全

| 等级 | 示例 | 要求 |
|---|---|---|
| 普通 | 通用模板、公开培训资料 | 可用于普通检索和问答。 |
| 内部 | SOP、流程模板、一般复盘案例 | 仅限授权人员查看和调用。 |
| 敏感 | 供应商表现、8D、批退记录、客户相关问题 | 限制导出和外发，必要时脱敏。 |
| 严格受控 | 客户保密资料、处罚评级、价格成本、重大事故、人员评价 | 默认不进公网模型，优先企业受控或本地模型。 |

## 10. 工程原则

1. 代码干净、可维护、灵活优先。
2. 命名清晰优先于注释。
3. 模块职责边界明确，避免业务模块互相引用。
4. TypeScript 严格模式。
5. 核心路径必须有自动化测试。
6. 新依赖必须有明确必要性。
7. 本地开发和内网演示优先简单可靠。
8. Prisma schema 变更后必须 generate / migrate，并重启 dev server 验证。

## 11. SPEC 验收标准

| 验收项 | 标准 |
|---|---|
| 工程可运行 | `npm run dev` 可启动，页面和 API 正常。 |
| 构建质量 | lint、typecheck、单元 / 集成测试、build 通过。 |
| 登录会话 | 登录、登出、受保护路由和 API 会话校验可用。 |
| 工作台聚合 | API 能返回我的任务、我的项目和项目推进摘要。 |
| 项目工作区 | 能读取项目结构、任务详情、动作权限和历史记录。 |
| 业务动作 | 完成、退回、关闭、不涉及、附件等动作状态和留痕正确。 |
| 权限隔离 | 不同角色无法访问或执行越权数据与动作。 |
| 可观测性 | 关键请求、错误和业务写操作可追踪。 |
| 模块边界 | NPQ 模块不污染全平台概念，后续模块可复用工作台能力。 |

# 供应链质量部 AI 综合工作台 PRD & SPEC

本文档分为两部分：

- **PRD：产品需求文档**，说明要做什么、给谁用、解决什么问题、包含哪些模块和页面。
- **SPEC：技术规格文档**，说明如何实现、采用什么架构、数据如何建模、权限和安全如何控制。

# Part 1. PRD：产品需求文档

## 1. 产品定位

供应链质量部 AI 综合工作台是一个面向供应链质量部的统一 AI 应用平台，用于聚合业务运行、组织管理、人才发展、知识资产、AI能力、工具技能和管理决策。

产品不是多个零散 AI 工具的集合，而是一个：

```text
统一入口
统一权限
统一数据底座
统一AI能力中心
统一工具技能层
统一审计与安全治理
```

的部门级 AI 工作台。

## 2. 产品目标

| 目标 | 说明 |
|---|---|
| 业务结果可见 | 在“我的工作台”展示三条主业务线的结果情况、风险卡点和我的关联事项。 |
| 日常待办聚合 | 将待处理、待复核、待确认、待补资料、待关闭事项集中到个人首页。 |
| 专项能力沉淀 | 各角色工作台承载专项拓展功能，如 NPQ 二三级计划生成、PQE PFMEA、SQE 8D初审。 |
| 知识可检索可复用 | 钉钉知识库作为知识源头，本地建立权限化索引和 RAG 检索。 |
| AI输出可复核 | AI做分类、汇总、判断建议、预警、生成，但关键结论必须人工确认。 |
| 管理决策可量化 | 提供业务、组织、人才三类管理看板，指标口径统一。 |
| 企业数据安全可控 | 按数据等级、权限、脱敏、模型路由、审计和应急机制控制数据风险。 |

## 5. 我的工作台需求

### 5.1 页面定位

“我的工作台”是每个业务用户的日常首页，不承载复杂拓展工具。它根据角色展示：

- 三条主业务线结果情况
- 我的业务待办
- 我的复核事项
- 我的风险提醒
- 我的专属业务信息
- 我的AI建议
- 我的知识推荐

专项拓展能力进入各角色工作台。

### 5.2 三条主业务线结果态势

| 主业务线 | 展示内容 |
|---|---|
| 新品质量策划 | 项目节点状态、TR阶段、风险清单、验证任务、历史问题复用情况。 |
| 量产品质管控 | 量产异常、复发风险、改善进度、跨厂区版本执行、过程影响。 |
| 供应商质量管理 | 供应商风险、8D状态、来料风险、供应模式差异、改善闭环。 |

### 5.3 角色专属日常信息示例

| 角色 | 我的工作台展示 |
|---|---|
| NPQ | 负责项目、TR节点、验证任务、历史问题复用、阶段材料待办。 |
| SQE | 负责供应商、8D状态、来料风险、供应商改善、相似案例。 |
| PQE | 量产异常、待确认分类、改善项、复发预警、节拍影响。 |
| EMS | 三地异常、版本偏差、厂区差异、代工厂改善、同步提醒。 |
| QCM | 检验任务、批次风险、标准版本、SI数据、判定建议。 |
| 实验室 | 测试任务、FAE分析、可靠性结论、ORT、标准评审输入。 |
| 部门负责人 | 全局风险、待审批、管理看板摘要、资源协调、AI管理简报。 |

## 10. MVP范围

> **更新于 2026-06-09 需求对齐。** MVP 范围调整为：一人开发、2-4 周交付、首期聚焦新品质量策划（NPQ）业务流。

### 10.1 MVP 交付项

| 模块 | MVP内容 | 优先级 |
|------|---------|--------|
| 项目管理看板 | NPI 项目列表 + TR 阶段视图（预置模板，项目可自定义）+ 进度状态总览。 | P0 必须 |
| 任务流转 | 三态流转（待处理→进行中→已完成），按项目+角色分配任务。 | P0 必须 |
| 用户认证 | 用户名+密码注册登录，基础会话管理。 | P0 必须 |
| 管理员后台 | 阶段模板配置 + 用户管理 + 使用情况看板。 | P0 必须 |
| 可观测性 | SQLite 事件存储 + JSON API（AI分析用）+ 使用情况看板（人看）。 | P0 必须 |

### 10.2 MVP 暂不包含

以下能力保留在长期路线图中，MVP 阶段不做：

| 模块 | 原因 |
|------|------|
| SQE/PQE/EMS/QCM/实验室工作台 | 第二、三条业务流后续接入 |
| AI 能力（分类、汇总、判断建议、风险清单生成等） | 架构预留入口，功能延后 |
| 知识管理 / RAG 检索 | 依赖钉钉知识库对接和 pgvector，延后 |
| 人工复核台 | 无 AI 输出时可暂不建设 |
| 复杂权限（ABAC、数据脱敏、审批流转） | MVP 仅做项目+角色基础权限 |
| 钉钉集成 | 无企业认证和 API 权限时暂不接入 |

## 11. PRD验收标准

| 验收项 | 标准 |
|---|---|
| 项目看板 | 能创建 NPI 项目、选择/自定义阶段模板、查看各项目阶段进度。 |
| 任务流转 | 能在项目内创建任务、分配给项目成员、流转状态（待处理→进行中→已完成）。 |
| 用户认证 | 能注册、登录、保持会话。不同用户看到自己参与的项目和分配的任务。 |
| 管理员后台 | 能管理用户、配置阶段模板、查看系统使用情况统计。 |
| 可观测性 | 系统事件写入结构化存储，JSON API 可供 AI 工具读取分析。 |
| 搭积木架构 | 管理员可在后台启用/停用功能组件，前端导航和路由自动响应。 |

> 以下为长期验收标准，MVP 阶段不适用但保留作为设计约束：

| 验收项 | 标准 |
|---|---|
| AI输出 | AI建议必须显示依据、来源、不确定点和复核入口。 |
| 知识检索 | 回答必须引用来源，不能召回无权限文档。 |
| 安全 | 敏感数据导出、模型调用、系统写回必须受控。 |

# Part 2. SPEC：技术规格文档

## 1. 技术基线

> **更新于 2026-06-09 需求对齐。** 单人独立开发，本地 Windows 环境，选型以轻量、低配置成本、易维护为原则。

```text
Frontend / Fullstack：Next.js App Router（全栈，前端+API 在同一工程）
UI：Tailwind CSS + shadcn/ui
Database(dev)：SQLite（零安装，单文件）
Database(prod)：PostgreSQL + pgvector（通过 Prisma 平滑迁移，改连接字符串即可）
ORM：Prisma
Workflow：建议 LangGraph.js（长期预留，MVP 阶段不实现）
RAG：本地文档解析 + chunk + embedding + hybrid retrieval + citation（长期预留）
Auth(MVP)：用户名 + 密码（NextAuth.js / 自定义 JWT）
Auth(长期)：企业登录 / 钉钉登录适配
AI Model：本地 Model Gateway，支持 OpenAI / Azure OpenAI / DeepSeek / Qwen / 本地模型切换（长期预留）
Deploy(dev)：本地 Windows，npm run dev
Deploy(path)：本地演示 → 同网络 IP 访问 → 内网服务器部署
CI：不设传统 CI 管道，代码质量靠 AI 工作流 + 本地 lint/test 保证
代码规范：TypeScript 严格模式 + ESLint + Prettier，核心模块有自动化测试
```

## 2. 系统架构

> **更新于 2026-06-09。** 采用单体全栈架构，内部按"平台层 → 业务流组件 → 功能组件"三层积木组织。共享平台能力（认证、权限、可观测性、UI）与业务流组件单向依赖。

### 2.1 三层积木架构

```text
系统平台（最大壳，统一入口/权限/数据/可观测性）
├─ 业务流：新品质量策划（NPQ）       ← 首期 MVP
│  ├─ 功能组件：项目管理看板
│  ├─ 功能组件：任务流转
│  ├─ 功能组件：风险清单             ← 长期
│  └─ ...
├─ 业务流：量产品质管控（PQE/EMS）   ← 后续接入
│  ├─ 功能组件：...
│  └─ ...
├─ 业务流：供应商质量管理（SQE）     ← 后续接入
│  ├─ 功能组件：...
│  └─ ...
└─ AI 能力嵌入各层（按需，不绑定到特定业务流）

管理员后台（独立模块）
├─ 阶段模板配置
├─ 用户管理
├─ 功能组件开关管理
└─ 可观测性面板
```

### 2.2 各层说明

| 层 | 职责 | 技术实现 |
|---|---|---|
| 平台层 `platform/` | 认证、权限（项目+角色）、可观测性（logging/metrics/tracing）、共享 UI 组件 | Next.js API Routes / Server Components |
| 业务流 `flows/<name>/` | 业务场景的项目管理、任务流转、阶段追踪、数据展示 | 依赖 platform，不可反向依赖 |
| 功能组件（业务流内部） | 可独立上下线的功能单元，由管理员在后台通过 ComponentConfig 表控制 | 常规路由 + 中间件检查启用状态 |
| 管理员 `admin/` | 模板配置、用户管理、组件开关、使用情况看板 | 独立路由组，仅管理员可访问 |

### 2.3 组件上下线机制

功能组件通过配置驱动开关，开发体验不受影响：

```text
ComponentConfig 表：{ id, name, path, enabled, description }

前端侧边栏：只渲染 enabled=true 的组件入口
后端中间件：检查请求路径对应的组件是否 enabled，否则返回 404
管理员后台：可视化开关，即时生效
```

### 2.4 跨域数据关联

业务流之间数据独立，但架构预留跨域关联能力：

- 统一主数据编码（项目编码、物料编码、供应商编码等）作为跨域关联键
- 各业务流的数据表预留 `externalSource`、`externalId`、`syncAt` 字段
- 未来可通过主数据编码或关联表实现跨业务流追溯

### 2.5 单体部署架构


## 5. AI能力中心

| 能力 | 实现方式 |
|---|---|
| 分类 | LLM + 枚举规则 + JSON Schema 校验。 |
| 路由 | 规则引擎优先，LLM 只给建议，最终由人确认。 |
| 汇总 | 多源数据查询 + LLM 结构化摘要。 |
| 判断建议 | 规则、历史案例、SOP、业务数据共同输入，输出依据和不确定点。 |
| 预警 | 定时任务 + 阈值规则 + 趋势识别 + AI摘要。 |
| 画像 | 聚合供应商、物料、产品、厂区、人员、知识资产历史数据。 |
| 追踪 | 工作流状态机 + 超期提醒 + 闭环进度。 |
| 检索推荐 | pgvector + 关键词过滤 + 权限过滤 + 来源引用。 |
| 结构化生成 | 固定模板 + LLM 填充 + 人工确认。 |

### 5.1 标准调用链路

```text
用户请求
→ UserContext 识别
→ 功能权限和数据权限判断
→ 数据等级识别
→ 字段脱敏或上下文裁剪
→ Model Gateway 模型路由
→ AI 调用
→ JSON Schema / 输出格式校验
→ 置信度、不确定点和引用来源生成
→ 高风险场景进入人工复核
→ 审计记录写入 ModelInvocation / AuditEvent
→ 展示、生成文件或受控写回
```

## 7. 钉钉知识库联动

```text
钉钉知识库作为原始知识源
→ 同步知识库、目录、文档元数据、权限、更新时间、来源链接
→ 可下载内容进入本地解析、分块、向量化
→ 用户检索时先按本地角色权限 + 钉钉权限过滤
→ AI回答必须带钉钉原文链接和引用来源
```

关键规则：

- 本地索引不能绕过钉钉权限。
- 钉钉文档权限变化后，本地索引必须同步更新或失效。
- 跨知识库检索必须同时满足本地权限和钉钉权限。
- 过期、废止、版本冲突文档不得参与AI回答。

## 8. 企业数据安全

### 8.1 数据分级

| 等级 | 数据示例 | 要求 |
|---|---|---|
| 普通 | 公开培训资料、通用方法论、无敏感字段模板 | 可进入知识库和AI问答。 |
| 内部 | SOP、检验标准、流程模板、一般复盘案例 | 仅限授权人员查看和调用AI。 |
| 敏感 | 供应商表现、8D、批退记录、客户相关问题 | 默认限制导出和外发，必要时脱敏。 |
| 严格受控 | 客户保密资料、供应商处罚/评级、价格成本、重大事故、人员评价 | 默认不进入公网模型，优先本地或私有化模型。 |

### 8.2 控制措施

| 控制项 | 要求 |
|---|---|
| 数据准入 | 业务对象、字段、知识文档必须标记敏感等级、责任人和可用范围。 |
| 权限过滤 | RAG、AI总结、文件生成前必须做权限过滤。 |
| 脱敏处理 | 客户、供应商、价格、人员评价、事故编号支持脱敏。 |
| 模型路由 | 按数据等级选择公网、企业受控、本地或私有化模型。 |
| 工具白名单 | 所有工具必须注册、授权、审计。 |
| 导出控制 | 敏感数据导出记录导出人、字段、脱敏状态、用途和审批状态。 |
| 审计留痕 | 记录AI输入摘要、检索来源、输出、工具调用、人工修改、最终动作。 |

## 9. 权限与治理

> **更新于 2026-06-09。** MVP 阶段采用轻量项目+角色权限模型，长期演进到 RBAC+ABAC。

### 9.1 MVP 权限模型：项目 + 角色

用户本身无固定系统角色。权限由"用户被加入哪个项目 + 在该项目中被赋予什么角色"决定。

```text
用户 ──被加入──> 项目（Project）
                  ├─ 角色：项目负责人（可编辑项目、分配任务、自定义阶段）
                  ├─ 角色：任务执行者（可查看项目、处理分配给自己的任务）
                  └─ 角色：观察者（只读查看项目进度）
```

核心规则：
- 用户只能看到自己参与的项目
- 用户只能看到分配给自己的任务（项目负责人可看全部）
- 管理员后台独立于项目权限，仅特定 Admin 角色可访问

### 9.2 长期权限模型（保留设计）

```text
RBAC：控制角色能进入哪些模块和功能
ABAC：控制用户能看到哪些数据、字段、知识和组件
```

权限维度：

- 角色
- 业务线
- 厂区
- 供应商
- 项目
- 流程节点
- 数据敏感等级
- 动作权限：查看、编辑、导出、调用AI、审批、写回系统

### 9.3 长期权限申请（保留设计）

```text
权限不足提示
→ 用户提交 AccessRequest
→ 系统带出申请对象、数据范围、敏感等级和建议有效期
→ owner / 管理员审批
→ 权限生效并记录有效期
→ 到期自动回收或进入复核
→ 全流程写入审计日志
```

## 10. 工作流建议

建议采用 LangGraph.js 承载状态化、可恢复、可人工中断/复核的业务闭环。

本方案不展开具体图结构，待各业务线完成0层梳理后，再按实际节点、责任人、判断规则、异常分支和系统接口逐项设计。

原则：

- 业务流程优先于 Agent 自主性。
- AI 只做辅助分类、汇总、判断建议、预警和生成。
- 关键质量结论、供应商处置、客户输出、人才评价必须人工复核。
- 所有关键节点保留状态、责任人、操作记录和AI输入输出。

## 10-1. 可观测性设计

> **新增于 2026-06-09 需求对齐。** 可观测性三件套（Logging + Metrics + Tracing）贯穿所有模块。存储于 SQLite，双出口设计：JSON API 给 AI 分析，看板页面给人看。

### 10-1.1 设计目标

- AI 易读性优先：结构化 JSON，AI 工具能直接读取分析，辅助排查问题
- 人可看：后台使用情况看板展示基础统计
- 零额外依赖：和业务数据共用 SQLite，Prisma 统一管理
- 贯穿全链路：每个 API 请求从入口到出口有唯一 traceId

### 10-1.2 事件类型

```text
ObservabilityEvent 表：
  id, traceId, spanId, parentSpanId,
  eventType: 'request' | 'db_query' | 'ai_call' | 'user_action' | 'error',
  path, method, userId, projectId,
  statusCode, durationMs,
  requestBody, responseSummary,
  errorMessage, errorStack,
  timestamp
```

### 10-1.3 双出口

| 出口 | 消费者 | 形式 |
|------|--------|------|
| JSON API `/api/observability/events?traceId=xxx` | AI 工具（Claude Code 等） | 结构化 JSON |
| 看板页面 `/admin/observability` | 人 | shadcn 图表（请求量、错误率、耗时分布） |

### 10-1.4 MVP 实现范围

- 每个 API 请求自动记录（middleware 注入）
- 核心业务操作（创建项目、流转任务）手动打点
- traceId 从前端传后端，贯穿完整调用链
- 使用情况看板：今日请求量、P50/P95/P99 延迟、错误率

| 可观测性 | API 请求自动写入事件表，traceId 贯穿调用链，JSON API 可查询，使用情况看板显示基础统计。 |

| 模块边界 | `flows/` 不反向引用 `admin/`，业务流之间无直接依赖。依赖方向符合设计。 |

### 12.2 长期验收（保留设计）

| 验收项 | 测试方式 |
|---|---|
| 业务对象完整性 | 抽查 SQE、PQE、NPQ、EMS、QCM、实验室是否都能映射到统一业务对象。 |
| 主数据可关联 | 用供应商、物料、批次、异常单、8D、改善项做完整链路关联测试。 |
| 指标口径一致性 | 检查 MetricDefinition 中公式、维度、周期、数据源、负责人和版本。 |
| 权限矩阵 | 用不同角色账号验证菜单、数据、导出、AI调用、审批权限差异。 |
| 数据安全 | 用敏感字段测试脱敏、模型路由、导出审批和审计记录。 |
| RAG权限过滤 | 用户无法通过知识问答检索到无权限钉钉文档内容。 |
| 钉钉知识同步 | 文档新增、更新、失效、权限变化后，本地索引能同步或标记重建。 |
| AI结构化输出 | 分类、汇总、8D初审、异常分析输出必须符合 JSON Schema。 |
| 工具调用审计 | 网页浏览、文件生成、钉钉通知、数据查询均有审计记录。 |
| 人工复核 | 高风险动作不能直接完成，必须进入复核台。 |
| 组件注册 | 每个组件有唯一标识、数据源、参数schema、权限策略、展示schema和刷新策略。 |
| 个人工作台 | 用户可复制模板、增删组件、拖拽布局、保存个人视图。 |
| 新功能扩展 | 新功能可通过组件注册加入组件库，并被个人工作台和团队看板引用。 |

## 13. 默认假设

> **更新于 2026-06-09 需求对齐。** 以下假设基于单人开发、本地 Windows 环境、2-4 周 MVP 交付的前提。

### 13.1 MVP 阶段假设

- 单人独立开发，所有决策以单人可以维护为前提。
- 工程优先：代码干净、可维护、灵活，宁可多花 30% 时间也不埋技术债。
- SQLite 用于 MVP 开发，零安装成本，Prisma 保证平滑迁移 PostgreSQL。
- Next.js App Router 全栈，一个工程搞定前后端，不拆多个项目。
- 用户名+密码认证即可，后续接企业认证。
- 先本地演示给同事看，再逐步扩展到同网络访问和正式部署。
- 新品质量策划为首条业务流，后续业务流以独立 flow 组件接入，共享平台能力。
- 数据模型一步到位为长期设计，MVP 阶段数据手动录入但表结构预留集成字段。
- 核心模块有自动化测试，但不对覆盖率设硬性指标。
- 所有核心能力优先本地代码实现，不强依赖 Dify 或外部平台。
- AI 功能（分类、汇总、风险清单生成等）架构预留入口，功能延后实现。

### 13.2 长期假设（保留）

- Dify 只作为可选参考、过渡工具或能力对比对象。
- 钉钉知识库作为知识源头，本地只做权限化索引、检索增强、AI引用和审计。
- 钉钉身份、部门和知识库权限需要与本地用户、角色和数据范围同步映射。
- 网页浏览、文件生成、系统查询、钉钉联动等能力通过工具与技能层受控调用。
- 企业数据安全模块贯穿所有工作台、AI能力、工具技能、知识库同步和系统集成。
- 敏感和严格受控数据默认不直接进入公网模型，必须经过权限判断、脱敏或模型路由控制。
- 权限设计依赖数据对象、字段口径、敏感等级和业务范围梳理结果。
- 复杂工作流仅保留架构建议，具体流程图在业务0层完成后单独设计。
- 初期外部系统接口不齐时，允许使用 Excel/CSV、手工录入和 API mock 过渡。
- 工作台和看板采用模块化组件方案，预设角色工作台只是模板，不是用户可用视图的上限。
- 所有正式管理指标必须先进入 MetricDefinition，再被看板组件引用。
- 权限不足必须通过 AccessRequest 申请和审批，不允许线下绕过系统长期授权。

## 14. 核心工程原则

> **新增于 2026-06-09 需求对齐。** 贯穿所有模块的开发铁律。

1. **代码干净可维护且灵活** — 最高优先级。宁可多花 30% 时间写干净的代码，也不为赶进度埋坑。
2. **命名清晰 > 注释** — 变量名、函数名、文件名自解释，注释只用于解释"为什么"而非"是什么"。
3. **模块职责边界明确** — 改一个模块不波及全局。`platform/`、`flows/`、`admin/` 之间有清晰的单向依赖。
4. **新人友好** — 新人（或三个月后的自己）打开代码能在一小时内理解核心结构。
5. **数据模型一步到位** — 表结构按长期设计，预留外部系统集成字段（`externalSource`、`externalId`、`syncAt`）。
6. **TypeScript 严格模式** — 所有代码通过 strict 检查，不偷懒用 `any`。
7. **自动化测试覆盖核心路径** — 项目创建、任务流转、权限判断、组件开关等关键逻辑必须有测试。
8. **单体但不耦合** — 一个工程，但内部结构按"将来能拆成独立包"组织。
9. **所有外部依赖受控引入** — 新增 npm 包必须有明确必要性，优先使用平台内置能力。
10. **不引入 Docker/K8s 等基础设施层** — MVP 阶段保持最简单部署形态，未来需要时再加。
